# v0.1 Implementation Plan

**Status:** Draft
**Date:** 2026-04-25
**Type:** Project plan
**Synthesizes:** ADRs 0002–0009 into concrete v0.1 work

## Purpose

This plan operationalizes ADR-0002 — Architecture Overview's v0.1 milestone — *connect, authenticate, list worlds* — into specific crates, dependencies, milestones, and acceptance criteria. It is the bridge between architectural intent and the first running daemon.

The plan deliberately scopes small. v0.1 is the foundation everything else builds on; getting the bones right matters more than reaching for in-world features that ADR-0002's roadmap places at v0.2 and beyond.

## v0.1 goals

The v0.1 server should let an admin or player:

1. Connect via telnet to a configured host and port.
2. See a connection banner identifying the server.
3. Authenticate against a real account store with hashed passwords.
4. See a list of available worlds.
5. Select a world and receive a placeholder confirmation ("you would now be in world X").
6. Disconnect cleanly, with the server logging the session lifecycle.

The server should run reliably under normal connection load, handle ungraceful disconnects, shut down cleanly on SIGINT/SIGTERM, and produce structured logs throughout.

## v0.1 non-goals

The following are explicitly **out of scope** for v0.1, even though some are tempting to include:

- **No in-world simulation.** No Cells, no Things, no movement, no `look`. World selection ends with a message and a disconnect. Per ADR-0002 these arrive in v0.2 and v0.3.
- **No soft-code.** No Lykn integration, no triggers, no scripted commands. Per ADR-0008, v0.4.
- **No description assembly.** No salience matching, no fragment tagging. Per ADR-0002, v0.5+.
- **No WebSocket protocol.** Telnet only for v0.1. WebSocket is a v0.2+ addition.
- **No GMCP-style structured side-channel.** Plain text protocol. Side-channel work begins when in-world simulation needs it (v0.2 or v0.3).
- **No builder client.** The ratatui TUI is its own track, separate from server v0.1.
- **No multi-world simulation.** The world registry exists and lists worlds, but no world's tick runs in v0.1 because there's no in-world state yet.
- **No internationalization.** English-only banners, prompts, and error messages.
- **No federation, clustering, or distributed deployment.** Single-process server.

The discipline here is sharp: anything that doesn't directly support the six v0.1 goals above is deferred. Scope creep at the foundation costs the most.

## User stories

**As an admin, I can stand up a server.** I configure the listen address, account store path, and world registry path in a TOML file. I run the binary. The server starts, logs its bound address, and begins accepting connections.

**As an admin, I can create accounts.** I run a CLI subcommand (`liaozhai-server account create <username>`) that prompts for a password, hashes it with argon2, and writes the account to the SQLite store.

**As a user, I can connect and authenticate.** I telnet to the server. I see a banner. I'm prompted for username and password. On success I'm shown a list of worlds. On failure I'm prompted to retry, with a finite retry count and per-IP rate limiting on failed attempts.

**As a user, I can see and select worlds.** After authentication, I see a numbered list of worlds with their names and short descriptions. I select one. I see a placeholder ("In v0.1, you would now be in world X. Disconnecting.") and the connection closes.

**As an operator, I can observe and shut down the server cleanly.** Logs are structured and adjustable via `RUST_LOG`. SIGINT or SIGTERM triggers a graceful shutdown that closes listeners, drains existing connections with a "server shutting down" message, and exits with code 0.

## Crate scaffold

Per ADR-0009 — Workspace crate layout, the v0.1 workspace contains five crates beyond the existing umbrella:

```
liaozhai/
├── Cargo.toml                      # workspace manifest
└── crates/
    ├── liaozhai/                   # umbrella (already exists, published 0.0.1)
    ├── liaozhai-core/              # NEW: shared types, IDs, error enum
    ├── liaozhai-net/               # NEW: telnet codec, session state machine
    ├── liaozhai-auth/              # NEW: account model, SQLite, argon2
    ├── liaozhai-worlds/            # NEW: world registry, world metadata
    └── liaozhai-server/            # NEW: binary; ties everything together
```

Crates that ADR-0009 names but v0.1 doesn't build yet (`liaozhai-world`, `liaozhai-cmd`, `liaozhai-persist`, `liaozhai-script`, `liaozhai-text`) are deferred to the version that needs them.

### Per-crate responsibilities for v0.1

**`liaozhai-core`** holds workspace-wide types: `AccountId(Uuid)`, `WorldId(Uuid)`, `ConnectionId(Uuid)`, the project-wide `Error` enum (with `thiserror`), shared constants (protocol version, banner template), and feature flags. Library only. No I/O.

**`liaozhai-net`** owns the telnet codec, the connection state machine (`Connecting → Authenticating → WorldSelection → Disconnected`), the line-oriented framing, and the per-connection task surface. Speaks bytes and frames; does not know about specific accounts or worlds. Exposes a trait or channel-based API that `liaozhai-server` wires into.

**`liaozhai-auth`** owns the account schema (SQLite via `rusqlite`), password hashing (`argon2`), authentication flow (with retry/lockout policy), and the account-creation API used by the CLI subcommand. Library; the CLI subcommand lives in `liaozhai-server`.

**`liaozhai-worlds`** owns `WorldMetadata` (id, slug, display name, short description, registry-level config), the TOML loader for `worlds.toml`, and the in-memory `WorldRegistry`. v0.1 has no actual world simulation — this crate only catalogues. v0.2 expands its scope to include `WorldHandle` and lifecycle.

**`liaozhai-server`** is the binary crate. Owns `main.rs`, the tokio runtime setup, configuration loading (`clap` + `toml`), the listener acceptance loop, wiring connections to the net+auth+worlds crates, and the `account` CLI subcommand. Also handles graceful shutdown.

## Dependency choices

Workspace-level dependencies, pinned as workspace deps:

```toml
[workspace.dependencies]
tokio              = { version = "1", features = ["full"] }
tracing            = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter", "fmt"] }
serde              = { version = "1", features = ["derive"] }
toml               = "0.8"
clap               = { version = "4", features = ["derive"] }
thiserror          = "2"
anyhow             = "1"
bytes              = "1"
uuid               = { version = "1", features = ["v4", "serde"] }
rusqlite           = { version = "0.31", features = ["bundled"] }
argon2             = "0.5"
rpassword          = "7"     # for password prompts in CLI
```

Notes on the choices:

**`tokio` over `async-std`:** the ecosystem assumes tokio. (See Rust skill, CA section.)
**`thiserror` in `liaozhai-core`, `anyhow` in `liaozhai-server`:** the standard library/binary split. (See Rust skill, EH section.)
**`tracing` over `log`:** structured events, span context, ergonomic instrumentation. (See Rust skill, LO section.)
**`clap` derive:** standard CLI parser. (See Rust skill, CLI-05.)
**`rusqlite` (bundled) over `sqlx`:** v0.1 has trivial query needs; SQLx's compile-time query verification is overkill. Wrap calls in `tokio::task::spawn_blocking` for async integration. SQLx remains an option if migrations or compile-time queries become valuable later.
**`argon2` over `bcrypt`:** modern recommendation; argon2id is the OWASP default for password hashing.

Test-time dependencies:

```toml
[workspace.dependencies]
# (added to dev-dependencies of relevant crates)
tokio-test         = "0.4"
insta              = "1"     # snapshot tests for protocol output
assert_cmd         = "2"     # CLI subcommand tests
predicates         = "3"     # CLI assertion helpers
```

## Milestones

Each milestone is independently demonstrable, has clear acceptance criteria, and depends only on prior milestones. Sized for 1–3 days of focused work; the full v0.1 arc is roughly 2–3 weeks at a part-time pace.

### M1 — Workspace skeleton + tokio listener

**Scope:** Create the five new crates with their `Cargo.toml` manifests inheriting from `[workspace.package]`. Wire `liaozhai-server` to a tokio runtime. Open a TCP listener on a configured port. Accept connections, send a banner, close the connection.

**Acceptance criteria:**

- `cargo build --workspace` succeeds with no warnings.
- `cargo run --bin liaozhai-server -- run --port 4444` opens port 4444.
- `telnet 127.0.0.1 4444` connects, sees a banner, and the server immediately closes the connection.
- `tracing` is wired; `RUST_LOG=info cargo run` prints structured logs of accept/close events.
- A `--config <path>` flag accepts a TOML file with a `[server]` section containing `bind_address` and `port`.

**Dependencies:** none.

**Notes:** This milestone establishes every crate's existence and the tokio + tracing + clap + serde scaffolding. The connection lifecycle is trivial; the milestone is about plumbing.

### M2 — Telnet codec + line-oriented input

**Scope:** Implement a tokio `Codec` for line-oriented telnet — reads bytes, splits on CRLF, returns lines as UTF-8 strings (with replacement on invalid bytes). Defensive handling of telnet IAC sequences (strip them; don't crash). An `echo` "command" — whatever the client types is sent back.

**Acceptance criteria:**

- `telnet 127.0.0.1 4444` connects; typed lines are echoed back.
- Lines longer than 4096 bytes are rejected with a polite error and the connection stays open.
- IAC byte sequences (e.g., from a client that auto-negotiates) don't break the session — they're stripped or ignored.
- Connection task lifecycle is clean: dropped connections release resources, no leaks under repeated connect/disconnect.

**Dependencies:** M1.

### M3 — Connection state machine

**Scope:** Replace the M2 echo with a real connection state machine: `Connecting` → `Authenticating` → `WorldSelection` → `Disconnected`. Each state has its own input handler and prompt. `Authenticating` is initially stubbed (any non-empty username/password succeeds); real auth lands in M4. World listing and selection are also stubbed (hardcoded list); real registry lands in M5. The placeholder "you would now be in world X. Disconnecting." message ends the flow.

**Acceptance criteria:**

- A scripted telnet session walks the full state machine: banner → username prompt → password prompt → world list → world selection → goodbye message → disconnect.
- Invalid inputs at each state produce useful error messages without advancing the state.
- A `quit` command is honored at any state; the session ends gracefully.
- State machine transitions are logged at `debug` level.

**Dependencies:** M2.

### M4 — Account auth: SQLite + argon2

**Scope:** Account schema in SQLite, account-creation CLI subcommand, real authentication replacing M3's stub. Argon2id password hashing. Per-IP rate limiting on failed login attempts (simple in-memory tracker; no persistence). Login retry counter (3 attempts, then disconnect).

**Acceptance criteria:**

- `cargo run --bin liaozhai-server -- account create alice` prompts for a password (twice, for confirmation) and writes a row to the accounts SQLite database.
- `cargo run --bin liaozhai-server -- account list` shows existing accounts (no passwords).
- Telnet login succeeds with correct credentials, fails with incorrect ones.
- After 3 failed attempts, the connection is closed.
- Repeated failed-login attempts from the same IP within 60 seconds are throttled (rate-limited).
- The accounts table schema is documented in a `docs/dev/schema.md` (or per-crate doc).

**Dependencies:** M3.

**Risks:** SQLite concurrency under tokio (mitigation: use `tokio::task::spawn_blocking`; document any pooling decisions). Argon2 parameters affecting login latency (mitigation: tune via config; document defaults).

### M5 — World registry + selection

**Scope:** `WorldMetadata` and `WorldRegistry` types. TOML loader for a `worlds.toml` file specifying available worlds. Replace M3's hardcoded world list with reads from the registry. World selection validates the chosen world exists.

**Acceptance criteria:**

- A `worlds.toml` file with three world entries is loaded at server startup; the count is logged.
- Telnet → login → see the three worlds listed with names and short descriptions.
- Selecting a valid world produces the placeholder message and disconnect.
- Selecting an invalid world (out of range, typo) re-prompts.
- The world registry is cleanly extensible — adding a fourth world to the TOML and restarting the server makes it available.

**Dependencies:** M4.

### M6 — Polish, tests, ship

**Scope:** Integration test harness (a tokio-based telnet client driver that scripts sessions). Unit test coverage for each crate. Per-crate documentation (top-level rustdoc on each crate's `lib.rs`). Updated workspace README pointing at how-to-run instructions. Graceful shutdown on SIGINT/SIGTERM. Connection limits (max concurrent connections, configurable). Tag v0.1.0.

**Acceptance criteria:**

- Test coverage on the four library crates (`liaozhai-core`, `liaozhai-net`, `liaozhai-auth`, `liaozhai-worlds`) is at or above 80% (95% is the project ideal but realistic v0.1 target is 80%).
- Integration test harness scripts the full acceptance demo (account create → telnet → login → world list → select → disconnect → server shutdown) and the test passes.
- `cargo doc --workspace --no-deps` produces complete docs without warnings.
- `cargo clippy --workspace -- -D warnings` is clean.
- `cargo fmt --all -- --check` is clean.
- SIGINT/SIGTERM trigger graceful shutdown: stop accepting new connections, send "server shutting down" to existing connections, drain for up to 10 seconds, exit cleanly.
- A `max_connections` config setting is honored; the 101st simultaneous connection (with default 100) gets a polite rejection.
- Workspace README has a "Quick start" section that walks a new contributor from clone to running the demo.
- Git tag `v0.1.0` is ready to be applied.

**Dependencies:** M5.

**Notes:** This milestone is largely operational hardening. Don't skip it; v0.1 stops being v0.1 work and starts being v0.2 work the moment we let polish slide.

## Test strategy

**Unit tests** live in each crate's `src/` modules (`#[cfg(test)] mod tests`). Focus on:

- `liaozhai-core`: type construction, error variant formatting, ID generation determinism with seeded RNG.
- `liaozhai-net`: codec parsing (line splitting, IAC stripping, max-length rejection), state machine transitions, reject-paths.
- `liaozhai-auth`: argon2 hash/verify roundtrip, schema migrations, retry-counter logic, rate-limit window behavior.
- `liaozhai-worlds`: TOML deserialization, registry validation (no duplicate ids, no duplicate slugs), selection by index.

**Integration tests** live in `tests/` directories at the workspace root and in `liaozhai-server`. They drive the running server end-to-end via a tokio TCP client. The integration harness:

- Starts the server on an ephemeral port.
- Connects a tokio TCP client.
- Drives the connection through scripted scenarios (correct login, wrong password, invalid world index, etc.).
- Asserts on banner content, prompt sequences, response timing.
- Tears down cleanly.

**Snapshot tests** (`insta`) capture protocol output (the server's user-visible text) so cosmetic changes are caught explicitly. The banner, world-list formatting, and error messages are good candidates.

**CLI tests** (`assert_cmd`) cover the `account create` and `account list` subcommands.

**Manual exploratory testing** is also valuable for v0.1 — telnet from real clients on Linux, macOS, and (via PuTTY or similar) Windows. Record any quirks.

## Configuration

Server configuration lives in a TOML file (default `./liaozhai.toml`, overridable with `--config`):

```toml
[server]
bind_address = "127.0.0.1"
port = 4444
max_connections = 100

[auth]
db_path = "./data/accounts.db"
argon2_memory_cost = 19_456     # 19 MiB; tune for hardware
argon2_time_cost = 2
argon2_parallelism = 1
max_login_attempts = 3
rate_limit_window_secs = 60
rate_limit_max_failures = 10

[worlds]
registry_path = "./data/worlds.toml"

[logging]
default_filter = "info"          # overridable via RUST_LOG
```

The world registry (`./data/worlds.toml`):

[[world]]
id    = "studio-dusk"

name  = "The Mountain Trail"

## Risks and unknowns

**Telnet quirks across clients.** Different telnet clients (BSD telnet, PuTTY, iTerm2, MUSH-specific clients like MUSHclient and Mudlet) handle IAC negotiation, line-ending conventions, and character encoding differently. *Mitigation:* keep our telnet handling defensive — strip IAC, accept either CRLF or LF, treat bytes as UTF-8 with replacement on errors. Test with at least three clients before tagging.

**SQLite + tokio integration patterns.** `rusqlite` is sync; mixing it with tokio requires `spawn_blocking`. Done sloppily this becomes a perf and correctness problem (long-held blocking tasks, lock contention). *Mitigation:* keep auth queries small (login = single SELECT + verify; create = single INSERT). Document the pattern clearly in `liaozhai-auth`'s rustdoc. Revisit with `sqlx` or a connection pool if v0.4 work surfaces problems.

**Argon2 calibration.** Default argon2 parameters need to balance security and login-latency. Too low: weaker against attack. Too high: noticeable login delay. *Mitigation:* default to OWASP-recommended parameters; expose them in config; document tuning.

**Cross-platform signal handling.** `SIGTERM` doesn't exist on Windows. *Mitigation:* use `tokio::signal::ctrl_c` for portable shutdown; conditionally wire `SIGTERM` on Unix via `tokio::signal::unix`. Test on Linux + macOS at minimum; Windows confirmation can come from a contributor.

**Banner aesthetic.** The CJK glyphs (`聊齋`) in the banner depend on the client's font/encoding. Some terminals will mojibake. *Mitigation:* offer an ASCII-only banner option in config; default to the CJK form. Document the option in the README.

**Test coverage tooling.** `cargo llvm-cov` is the modern choice; CI integration is straightforward but the first setup takes care. *Mitigation:* land basic coverage measurement in M6, leave 95%+ enforcement for v0.2 hardening if v0.1 budget runs out.

## Definition of done for v0.1

The project ships v0.1.0 when all of the following hold:

- All six milestones' acceptance criteria pass.
- The acceptance demo (above) runs without error or warning on Linux and macOS.
- `cargo test --workspace` is green.
- `cargo clippy --workspace -- -D warnings` is clean.
- `cargo fmt --all -- --check` is clean.
- `cargo doc --workspace --no-deps` is clean.
- The README's quick-start works for a fresh clone.
- The `v0.1.0` git tag is annotated with release notes.
- The umbrella `liaozhai` crate at v0.1.0 is published to crates.io (or held back deliberately, with rationale).

## What's deferred — and where it goes

To keep v0.1 honest, here's where the eager-eyed-but-out-of-scope items go:

- **Look at a Cell, walk between Cells:** v0.2.
- **Builder-side world construction (create Cells, set descriptions):** v0.3.
- **Things, attributes, basic interactivity, Lykn integration:** v0.4.
- **Salience-based description assembly:** v0.5.
- **Píngdiǎn technique components:** v0.6+.
- **WebSocket protocol:** v0.2 (or earlier if a client demands it).
- **GMCP-style structured side-channel:** when in-world simulation needs it (v0.2 or v0.3).
- **Builder TUI (ratatui):** its own track, beginning when in-world building is real (v0.3+).
- **Web client:** post-v1.0.
- **Federation, distributed deployment:** post-v1.0.

Future ADRs will refine each as their version arrives.
